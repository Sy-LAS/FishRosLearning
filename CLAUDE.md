# CLAUDE.md

本文件为 Claude Code 在此仓库中工作时提供行为指导。项目技术栈：**Python**（Google 规范）、**C++17**（Google 规范）、**CMake** 构建系统。

# 编写要求
## 1. 先思考再编码

**不要臆断。不要掩盖困惑。暴露权衡。**

实现之前：
- 明确陈述你的假设。如有不确定，直接提问。
- 若存在多种理解方式，全部列出 —— 不要默默选择一种。
- 若有更简单的方案，指出来。必要时拒绝过度设计。
- 若有不清楚的地方，停下来。说清楚哪里困惑，然后提问。

## 2. 简洁优先

**用最少代码解决问题。零推测性代码。**

- 不写超出需求范围的功能。
- 单次使用的代码不加抽象层。
- 不添加未被要求的「灵活性」或「可配置性」。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行但 50 行就能解决，重写它。

自问：「一个资深工程师会觉得这过度复杂吗？」如果是，简化。

## 3. 精准修改

**只动必须动的。只清理自己造成的烂摊子。**

编辑已有代码时：
- 不要顺手「优化」相邻代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配已有代码风格，即使你更倾向另一种写法。
- 若注意到无关的死代码，提出来 —— 但不要擅自删除。

当你的改动产生了孤立代码：
- 删除因你改动而不再使用的 import / 变量 / 函数。
- 不要删除改动前就存在的死代码，除非被明确要求。

检验标准：每一处改动都应能追溯回用户的具体需求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- 「加验证」→「先为无效输入写测试，再让测试通过」
- 「修 bug」→「先写能复现的测试，再让测试通过」
- 「重构 X」→「确保重构前后测试全部通过」

多步骤任务先陈述简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强成功标准让你能独立迭代。弱标准（「把它做出来」）需要不断追问澄清。

## 5. 项目技术约定

以下为本项目的具体技术规范：

**Python 代码**（Google Python Style Guide）：
- Python 3.10+，所有公开函数含类型注解与 docstring（Google 风格）
- 仅用标准库 + 已安装的 `pypdf`、`pdfplumber`、`fitz`（PyMuPDF）、`Pillow`
- 最大行宽 100 字符，静态检查：`pylint` + `mypy`

**C++ 代码**（Google C++ Style Guide）：
- C++17，文件名 snake_case，类名 PascalCase，方法名 snake_case
- RAII 资源管理，禁止裸 `new`/`delete`，全面 `const` 正确性
- 头文件用 `#pragma once`

**CMake 构建**：
```bash
cd cpp && mkdir build && cd build && cmake .. && make
```
- 依赖：Poppler、MuPDF、libcurl、yaml-cpp、fmtlib（均为系统级库）
- C++ 源文件必须配 `CMakeLists.txt`，不允许孤立 `.cpp` 文件
